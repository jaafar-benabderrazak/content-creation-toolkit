1. A module-level constant `ENV_VAR_MAP` — a dict mapping dotted config field paths to env var names:
   ```python
   ENV_VAR_MAP: dict[str, str] = {
       "notify.discord_webhook_url": "NOTF_DISCORD_WEBHOOK_URL",
       "notify.slack_webhook_url":   "NOTF_SLACK_WEBHOOK_URL",
       "suno.api_key":               "SUNO_API_KEY",
   }
   ```
   Note: OPENAI_API_KEY and REPLICATE_API_TOKEN have no corresponding PipelineConfig fields in the current schema — do NOT invent fields. Map only what exists in the Pydantic models.

2. A `ConfigProvenance` TypedDict (import from typing, use TYPE_CHECKING guard for Python <3.11 compat — just use a plain dict type hint at runtime):
   ```python
   # ConfigProvenance: dict[str, Literal["env", "yaml"]]
   # Using plain dict to avoid TypedDict import complexity
   ```

3. A classmethod `load_with_env_defaults(cls, path) -> tuple[PipelineConfig, dict[str, str]]` on PipelineConfig:
   - Calls `from_yaml(path)` to get the base config
   - Iterates ENV_VAR_MAP; for each entry:
     - Resolve the dotted path to the current field value (e.g. `config.notify.discord_webhook_url`)
     - If the field value is None or empty string (""):
       - Read env var via `os.environ.get(env_var_name)`
       - If env var is set and non-empty: apply it with `object.__setattr__` on the sub-model (Pydantic v2) or direct assignment (Pydantic v1 — model has `model_config = ConfigDict(frozen=False)` already, if not add it)
       - Record `provenance[dotted_path] = "env"`
     - Else: record `provenance[dotted_path] = "yaml"`
   - Returns `(config, provenance)`

   Implementation note for field assignment on frozen Pydantic models: use `model.model_fields_set` awareness. The safest approach is to rebuild the sub-model after mutation. For NotifySettings:
   ```python
   updated_notify = config.notify.model_copy(update={"discord_webhook_url": env_val})
   config = config.model_copy(update={"notify": updated_notify})
   ```
   Use `model_copy(update=...)` (Pydantic v2). Add a `try/except AttributeError` fallback using `.copy(update=...)` for Pydantic v1.

4. The existing `_load_api_key_from_env` validator on SunoSettings already handles SUNO_API_KEY at model construction time. `load_with_env_defaults` should still record provenance for `suno.api_key` by checking whether the YAML had a value before construction.    - Then call `from_yaml(path)` normally (the validator handles the actual injection)
